Parallel Face
Scanning

Use every core the Mac has during a scan, and show matches as they're found instead of only at the end of each video.

Design approved 2026-05-28 Apple Silicon · 16GB+

The shift

Today, the scan runs on a single core. On an 8-core M-series Mac that means roughly 12% of the machine's CPU is doing the work while the other 88% sits idle. This change spreads the workload across all available performance cores and adds live feedback to the UI while a scan is running.

Scope

✓ In scope

  • Parallel face scanning across all available cores
  • Live frame counter shown during a scan
  • Matches streamed into the UI as they're found, not in one batch at the end
  • Parallel reference loading (same change, same win for big reference folders)

✕ Not in this change

  • GPU acceleration
  • Distributed / multi-machine scanning
  • Migration to a queue-based job system (deferred until the product becomes SaaS)
  • Smart frame extraction by scene cuts (separate backlog item)

Each worker loads the face-detection model once and reuses it for every frame it handles. Frames come back as they finish — the UI updates the moment a match is confirmed.

Live progress (the visible upgrade)

What the editor sees

A "frame X of Y" counter ticks up during each video, and matches now appear in the results panel in real time — no more waiting for the whole video to finish to know if a person was found.

The matches stream into the existing results panel as the scan progresses. The editor can already start eyeballing matches while later parts of the timeline are still being processed.

Two safety nets

Snappy scans stay snappy

Spinning up worker processes takes a couple of seconds. For very short scans (under ~30 frames) that overhead would actually make things slower, so those small jobs keep using the existing single-process path. The user never notices a regression on quick clips.

One-cycle feature flag

The whole parallel path is behind a switch that can be flipped off without redeploying code. It stays in place for one development cycle so a real wedding-length job can validate the change in production-like conditions. Once validated, the flag is removed.

Reference loading rides along

The "first-time clustering after pointing at a big folder" pain has the same root cause and the same fix. It's included because the architectural change is shared — no extra design cost, immediate user-visible win on large reference folders.

How we know it works

  • Equivalence test. The same small fixture video is scanned with the new parallel path and with the old single-process path. Results must be identical. This is the primary guarantee against accuracy regressions.
  • Progress contract. The frame counter must reach the total, and the matches reported live must equal the final result set. No silent drops.
  • Benchmark run. A one-shot script measures wall-clock time and frames per second on both paths, so the speedup is documented rather than assumed.
